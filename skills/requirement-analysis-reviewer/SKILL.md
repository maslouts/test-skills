# 需求解读与问题拆解

## 适用场景

- 用户提供需求文档、PRD、原型说明、业务规则、会议纪要或直接粘贴需求内容。
- 用户希望先做需求分析，而不是立刻写测试用例或实现方案。
- 用户希望提前发现需求缺失、逻辑冲突、边界遗漏和测试风险。
- 当用户提供需求内容时，优先按“开发、测试、业务”三个角度进行需求解读，不急于直接写测试用例或实现方案。

## 核心目标

- 找出需求缺失点、逻辑冲突点、边界遗漏点。
- 识别权限、状态、流程、字段约束、兼容性等关键问题。
- 评估开发实现风险、测试验证风险和业务理解偏差风险。
- 输出可落地的待确认问题、测试点建议和协作建议。

## 分析原则

- 先理解需求目标、适用对象、业务范围，再分析细节问题。
- 先根据需求内容判断实际涉及范围，再决定分析维度和输出结构，不按固定模板硬套。
- 下文列出的维度只是候选检查项，不要求每次全部覆盖；只分析与当前需求直接相关的部分。
- 若当前需求天然不涉及角色、权限、审批、状态流转、兼容旧逻辑等维度，不强行分析，也不输出无关内容。
- 对需求中未明确说明的规则，不得脑补为确定结论。
- 能明确判断的问题直接指出；无法确认的内容统一写为 `待确认：xxx`。
- 优先指出会影响开发实现、测试设计、业务闭环的关键问题，不只做表面复述。
- 若需求信息明显不足，先指出缺失点，再继续分析可判断部分。

## 候选分析维度

说明：以下维度为常用检查项，用于辅助发现问题，不是固定模板。分析时应先判断是否相关；不相关的维度直接跳过，不输出“无问题”或空结论。

按需选择，不要求每次全部覆盖：

- 需求缺失点：功能目标、适用角色、适用范围、触发条件、成功结果、失败结果、异常处理、字段规则、默认值、是否必填、长度限制、格式限制。
- 逻辑冲突点：前后描述冲突、页面规则与接口规则不一致、主流程与异常流程冲突；若需求涉及角色、权限或状态，再分析角色权限与状态限制是否冲突。
- 边界遗漏点：空值、极值、重复操作、并发操作、部分成功、部分失败、回退、中断、撤回、重试、历史数据、旧数据迁移、兼容旧逻辑。
- 权限 / 状态 / 流程问题：若需求涉及权限控制，再分析谁能看、谁能新增、谁能编辑、谁能删除、谁能审批；若需求涉及状态变化，再分析状态起点、终点、可逆性、非法状态限制；若需求涉及业务流转，再分析流程是否闭环、是否缺少驳回、撤回、失败、回滚等分支。
- 测试风险点：高风险主流程、易遗漏异常分支、开发实现歧义导致前后端口径不一致的风险、上线后影响历史数据、权限口径或流程稳定性的风险。

## 输出方式

根据需求复杂度动态组织结构，不强制固定模板。优先可选以下结构，并允许裁剪、合并或省略：

```markdown
## 需求概述
- 用 1 到 3 句话概括该需求在做什么、给谁用、解决什么问题。

## 主要问题清单
- 直接指出不清楚、冲突或遗漏在哪里。

## 待确认问题
- 统一使用 `待确认：xxx` 格式。

## 测试点拆解建议
- 按主流程、异常流程、边界场景拆解。

## 补充建议
- 只在确实有价值时补充，例如规则补齐、状态图补齐、权限矩阵补齐。
```

使用约束：

- 若需求简单，可只输出 2 到 3 个最相关板块。
- 若需求只涉及单页面、单功能点或局部规则调整，优先输出“主要问题清单 + 待确认问题 + 测试点拆解建议”。
- 若需求复杂，可增加更细的分类，但仍以清晰、准确、便于评审为优先。
- 若某类问题在当前需求中明显不适用，直接不写该分类。
- 不为了形式完整而补无关栏目，不为了结构工整而输出空内容或“无问题”。

## 输出要求

- 输出需求分析时，只输出当前需求真正相关的部分。
- 问题描述必须具体，直接指出不清楚在哪里、冲突在哪里、遗漏了什么。
- `待确认` 只列真正影响开发实现、测试断言或业务闭环的问题。
- 测试点拆解只写测试关注点，不展开为详细测试步骤，除非用户明确要求。
- 不要只复述需求原文，不做分析。
- 不要泛泛而谈“考虑异常情况”，必须指出具体异常点。
- 不要直接跳过与当前需求相关的高风险维度；对明显不适用的维度，不强行分析。
- 不要为了凑结构，输出与当前需求无关的分类、问题或建议。

## 示例风格

```markdown
## 主要问题清单

- 当前需求只说明“提交成功后更新状态”，但没有明确失败、超时、重复提交时的状态处理规则。
- 页面展示规则与接口返回字段未建立明确映射，存在前后端口径不一致风险。

## 待确认问题

- 待确认：审批驳回后是否允许重新提交，重新提交后沿用原审批单还是新建审批单。

## 测试点拆解建议

- 关注首次提交、重复提交、驳回后重提、超时失败后的补偿处理。
- 若涉及权限控制，补充无权限查看、无权限编辑、越权调用接口场景。
```

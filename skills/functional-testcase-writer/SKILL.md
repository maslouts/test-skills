# 功能测试用例编写

## 适用场景

- 用户提供需求文档、PRD、原型、功能说明，要求生成功能测试用例。
- 用户提供现有页面、现有功能或交互表现，要求整理测试点或测试用例。
- 用户要求将结果默认落地到项目 `testcases/` 目录。

## 执行规则

1. 先判断任务类型：
   - 有需求文档/PRD/功能说明：按需求驱动方式生成。
   - 无正式需求，仅基于现有功能或页面：按功能观察驱动方式生成。
2. 若可写本地文件且用户未明确要求网页端直接输出：
   - 默认写入 `testcases/{需求名称}-测试用例-{序号}.md`。
   - 序号按同名前缀文件自动递增，格式为 `01`、`02`、`03`。
   - 完成后仅简要告知文件路径。
3. 若明确是网页端使用提示词，或当前环境无法写文件：
   - 直接输出 Markdown 测试用例正文。

## 信息不足时的处理

- 需求不完整但仍可产出部分可断言内容时，继续生成可明确部分。
- 无法确认的业务规则、权限、状态流转、限制条件、默认值、错误提示、联动逻辑、后端处理逻辑，统一写为 `待确认：xxx`。
- 不得自行补充未明确的确定性业务结论。
- 若缺失信息已导致核心场景无法形成可断言结果，则单独列出待确认问题，不强行生成不可靠用例。

## 输出结构

优先使用以下层级，按需求复杂度裁剪：

```markdown
# 文档主标题
## 功能模块名称
### 功能测试点
#### 验证点
##### 用例场景

- 具体、可验证的结果描述
```

约束：

- 不在最后一层场景节点后新增标题层级。
- 不输出 `###### 预期结果` 标题。
- 预期结果直接写在最后一层场景节点下方，用列表表达。
- 每条场景必须对应明确、可断言的结果。
- 不输出测试步骤。

## 场景编写要求

- 一条最后一层场景节点代表一条完整测试用例。
- 场景标题优先包含：操作主体、核心行为、关键输入数据；必要时补充前置条件或关键状态。
- 不使用编号。
- 不使用“正常场景”“异常场景”等笼统标题。
- 复杂需求优先从业务链路拆分，而不是机械按字段逐个罗列。

示例：

```markdown
##### 用户将库存为 2 的商品加入购物车并输入数量 5

- 商品添加失败。
- 系统提示“商品库存不足，当前库存为 2 件”。
- 购物车商品数量和总金额保持不变。
```

## 预期结果要求

- 必须具体、可验证、可断言、可复核。
- 可从页面提示、数据变化、状态变化、系统行为、权限限制、联动影响等角度描述。
- 不使用“应该”“可能”“正常显示”“功能正常”“结果正确”等模糊表述。

## 设计覆盖要求

生成时必须综合覆盖以下维度，但不需要显式写出方法名：

- 等价类：有效输入、无效输入。
- 边界值：最小值、最大值、临界值、空值、极限长度。
- 判定表：多条件组合影响结果的关键场景。
- 场景法：主流程、异常流程、回退、中断、重复操作。
- 错误猜测：非法输入、越权、重复提交、状态异常、重复点击。
- 状态迁移：状态变化前、中、后及非法状态下的操作限制。

## 质量要求

- 不遗漏需求中明确说明的功能点。
- 在不虚构业务规则的前提下覆盖正常、异常、边界、反向、权限、状态、流程、数据结果等维度。
- 各用例尽量相互独立，不依赖执行顺序。
- 描述保持专业、客观、工程化，便于执行、评审和沉淀。
- 不附加与测试无关的说明性文字。

## 文件命名

- 文件名保持简洁明确，适合作为用例资产名称。
- 避免使用 `/ \\ : * ? " < > |`。
- 建议控制在 30 个字符以内。

示例：

```text
testcases/购物车功能-测试用例-01.md
```
